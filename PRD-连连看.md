# PRD - Web 端连连看游戏

**版本：** v0.1 MVP  
**日期：** 2026-05-16  
**状态：** 待评审

---

## 1. 一句话定位

一款运行在浏览器端的经典连连看游戏，面向想打发碎片时间的休闲玩家，提供开箱即玩、无需下载的消除体验。

---

## 2. 目标用户与场景

**目标用户**

- 主要用户：18–40 岁，熟悉浏览器，偶尔需要放松的上班族或学生
- 次要用户：对经典小游戏有情怀的中老年用户（操作应足够简单）

**典型场景**

| 场景 | 描述 |
|------|------|
| 碎片时间填充 | 等公交、午休 10–15 分钟，不想安装 App，直接打开网页玩 |
| 桌面摸鱼 | 办公室用 PC 浏览器，单局 5–10 分钟快速消遣 |
| 怀旧复玩 | 曾经玩过 QQ 游戏大厅等平台连连看的用户，想重温经典规则 |

**当前替代方案**

- 下载安装端游客户端（门槛高）
- 各类小程序或 H5 游戏（广告多、体验粗糙）
- 搜索引擎找到的各类老旧 Flash 遗产（已无法运行）

---

## 3. 核心价值主张

- **零门槛**：打开网页即玩，无需注册、无需安装
- **规则纯正**：严格还原经典两折角连线规则，无魔改
- **无广告干扰（MVP 阶段）**：专注游戏体验本身，不被强制广告打断
- **响应迅速**：纯前端实现，无网络延迟影响游戏判定

---

## 4. 功能清单（MVP）

优先级说明：P0 = 上线必须有 / P1 = 强烈建议首版包含 / P2 = 二期再做

### P0 核心玩法

| # | 用户故事 | 备注 |
|---|----------|------|
| P0-1 | 作为玩家，我想看到一个随机生成的图案棋盘，以便开始一局游戏 | 默认 8×8 格，图案种类 ≤ 16 种，保证初始局面可解 |
| P0-2 | 作为玩家，我想点击一个图案将其选中（高亮），以便准备配对 | 再次点击同一格取消选中 |
| P0-3 | 作为玩家，我想点击第二个相同图案后系统自动判断连通性，以便知道是否消除成功 | 折角 ≤ 2、路径不穿过已有图案块 |
| P0-4 | 作为玩家，我想看到消除时两格之间的连线动画，以便获得即时反馈 | 线段沿路径显示后消失，耗时约 300ms |
| P0-5 | 作为玩家，我想在选错时看到抖动/红色提示，以便知道本次配对无效 | 无效点击不扣分、不计时 |
| P0-6 | 作为玩家，我想在全部图案消除后看到"通关"提示，以便知道游戏结束 | 显示用时和得分 |
| P0-7 | 作为玩家，我想点击"重新开始"按钮，以便立刻开始新的一局 | 重置棋盘和计时器 |

### P1 体验增强

| # | 用户故事 | 备注 |
|---|----------|------|
| P1-1 | 作为玩家，我想看到倒计时（如 5 分钟），以便增加紧迫感和挑战性 | 超时弹出"游戏结束"，显示剩余图案数 |
| P1-2 | 作为玩家，我想看到当前得分实时更新，以便了解自己的进度 | 每次消除加分，连消加成 |
| P1-3 | 作为玩家，我想使用一次"提示"功能，以便在卡关时找到一对可消除的图案 | 每局限用 3 次，高亮提示约 2 秒 |
| P1-4 | 作为玩家，我想使用一次"洗牌"功能，以便在无路可走时重排棋盘 | 每局限用 2 次，保证洗后有解 |
| P1-5 | 作为玩家，我想听到消除音效，以便获得更好的游戏体验 | 可在设置中关闭音效 |

### P2 二期功能（本期不做）

| # | 用户故事 |
|---|----------|
| P2-1 | 作为玩家，我想选择不同难度（棋盘尺寸：6×6 / 8×8 / 10×10），以便匹配自己的水平 |
| P2-2 | 作为玩家，我想查看历史最高分排行榜，以便与自己过去的记录比较 |
| P2-3 | 作为玩家，我想选择不同图案主题（动物、水果、麻将牌等），以便个性化体验 |
| P2-4 | 作为玩家，我想分享本局得分截图到社交平台，以便与朋友炫耀 |

## 5. 核心流程

```
打开网页
  └─> 展示主界面（游戏标题 + 棋盘 + 计时器 + 得分 + 提示/洗牌按钮）
        └─> 棋盘随机生成并渲染
              └─> [玩家点击图案 A]
                    ├─> 图案 A 高亮选中
                    └─> [玩家点击图案 B]
                          ├─> B 与 A 不同类型 → 红色抖动提示，保持 A 选中状态
                          ├─> B 与 A 相同类型，连通性检测失败 → 抖动提示，保持 A 选中
                          └─> B 与 A 相同类型，连通性检测通过
                                └─> 绘制连线动画（约 300ms）→ 消除 A、B → 更新得分
                                      ├─> 棋盘仍有图案且计时未结束 → 继续等待点击
                                      ├─> 棋盘清空 → 弹出"恭喜通关"弹窗（用时 + 得分 + 重玩按钮）
                                      └─> 剩余图案但无路可走（自动检测）→ 提示"无法继续"，建议洗牌
```

**连通性判断算法要点（供开发参考）：**
1. 两格之间最多允许 2 次折角（即 1、2 或 3 段线段）
2. 连线路径上不能经过任何未消除的图案格
3. 连线可以沿棋盘外围边缘走（即允许走棋盘外的空白边界）

## 6. 数据与状态

### 客户端状态（无需后端，纯前端）

| 数据项 | 类型 | 说明 |
|--------|------|------|
| `boardState` | 二维数组 | 当前棋盘每格的图案 ID（0 表示已消除） |
| `selectedCell` | {row, col} \| null | 当前选中的格子坐标 |
| `score` | number | 本局得分 |
| `timeLeft` | number | 倒计时剩余秒数 |
| `hintCount` | number | 剩余提示次数（初始 3） |
| `shuffleCount` | number | 剩余洗牌次数（初始 2） |
| `gameStatus` | enum | playing / cleared / timeout / no_moves |

### 本地持久化（localStorage，P1）

| 数据项 | 说明 |
|--------|------|
| `bestScore` | 历史最高分（单设备） |
| `soundEnabled` | 音效开关偏好 |

### 账号体系

MVP 阶段**无需账号**，完全匿名使用。历史记录仅存本地，清除缓存即丢失。

---

## 7. 不做什么

| 排除项 | 原因 |
|--------|------|
| 用户注册/登录 | MVP 阶段增加摩擦，且无需跨设备同步 |
| 多人联机/对战模式 | 需要后端支撑，复杂度远超 MVP |
| 移动端原生 App | Web 已可满足需求，App 需发布审核流程 |
| 道具商城/内购 | 变现不在本期目标内 |
| 可自定义图案上传 | 增加审核和存储复杂度 |
| 全球排行榜 | 需要后端和防刷机制，二期再做 |
| 无障碍/键盘操作 | 连连看天然依赖鼠标/触控，优先级低 |

## 8. 成功标准

上线后 **4 周内**，以下指标达标视为 MVP 验证通过：

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 单局完成率 | ≥ 50%（开始游戏后打到通关或超时） | 前端埋点事件 |
| 次日留存率 | ≥ 20% | localStorage 记录首次访问日期 |
| 平均每次会话局数 | ≥ 1.5 局 | 重新开始按钮点击次数/UV |
| 提示功能使用率 | ≤ 60%（说明难度适中，不依赖提示） | 提示按钮点击埋点 |
| 页面加载时间 | ≤ 2 秒（4G 网络） | Lighthouse / 手工测试 |

## 9. 开放问题

| # | 问题 | 影响范围 | 建议决策时间 |
|---|------|----------|--------------|
| Q1 | 图案素材用 Emoji、像素图还是矢量图标？Emoji 跨平台渲染差异大，像素图有情怀但需美术 | UI 风格、开发成本 | 开发启动前 |
| Q2 | 初始棋盘生成算法：纯随机后校验可解性，还是从已解状态反向打乱？两者性能和体验差异需验证 | 核心玩法稳定性 | 技术方案阶段 |
| Q3 | 无路可走时是否强制自动洗牌（消耗次数），还是仅提示让玩家手动触发？ | 游戏节奏感 | 原型测试后决定 |
| Q4 | 计时器是否需要在用户切换 Tab 时暂停？（防止超时惩罚不公平） | 游戏公平性 | 开发阶段确认 |
| Q5 | 移动端竖屏适配是否纳入 MVP？棋盘在小屏上的点击精度是关键风险 | 用户范围 | 尽早决定，影响布局方案 |
